# خلاصه‌ی اقدامات — مدیریت کاربران + بازطراحی پنل

> سند مرجع تاریخی این سشن کاری. کل feature تمام و روی `master` push شده؛ این فایل برای این نوشته شده که هرکس بعداً بخواد بفهمه چه اتفاقی افتاده، بدون نیاز به خود سشن.

**تاریخ:** ۲۰۲۶-۰۷-۱۸ تا ۲۰۲۶-۰۷-۱۹
**نتیجه‌ی نهایی:** روی `master`، کامیت‌های `94c36ae` تا `2e5e221` (۷ کامیت، ۱۰ فایل، ~۳۶۰۰ خط تغییر).

---

## ۱. هدف اولیه

امکان **مدیریت کاربران** در پنل ادمین پروژه‌ی v2ray-sub-auto: ساخت کاربر با مدت اشتراک مشخص، لینک اشتراک یکتا برای هرکس، و وقتی اشتراک منقضی می‌شه، به‌جای کانفیگ‌های واقعی فقط یک کانفیگ ساختگی («اشتراک شما به پایان رسیده») به کلاینت برگرده.

در کنارش، تصمیم گرفته شد پنل ادمین (که یک اسکرول ۲۴۰۰+ خطی شلوغ بود) بازطراحی بصری هم بشه.

---

## ۲. معماری و تصمیم‌های کلیدی

- **همه‌ی کاربران از یک استخر کانفیگ سراسری مشترک تغذیه می‌شن** (این پروژه aggregator است، نه هر کاربر کانفیگ اختصاصی دارد).
- **فعال‌سازی در اولین استفاده (Activation on First Use):** موقع ساخت کاربر تاریخ انقضا خالیه؛ اولین باری که لینک کشیده بشه، شمارش شروع می‌شه. این کار به‌صورت **اتمیک** (یک UPDATE با شرط) انجام می‌شه تا race condition نداشته باشیم.
- **وضعیت محاسبه‌ای، نه ذخیره‌شده:** ستون `status` فقط نیت ادمین رو نگه می‌داره (`ACTIVE`/`PAUSED`/`DISABLED`). حالت `EXPIRED` همیشه در لحظه‌ی خواندن از روی `expire_at` محاسبه می‌شه تا هیچ‌وقت دو منبع حقیقت نداشته باشیم.
- **توقف منصفانه (Pause/Resume):** وقتی کاربر Pause می‌شه، زمان باقیمانده منجمد می‌مونه؛ موقع Resume، دقیقاً همون مدت توقف به تاریخ انقضا اضافه می‌شه.
- **مدت `0` = نامحدود** (بدون تاریخ انقضا) — این یک تصمیم بعدی بود که در میانه‌ی کار اضافه شد.
- **حداکثر دستگاه:** فیلدی در دیتابیس هست (پیش‌فرض ۳، `0`=نامحدود) ولی enforce نمی‌شه — چون این پروژه یک aggregator است و enforcement واقعی (چند دستگاه هم‌زمان) غیرقابل‌اعتماد است.
- **لینک عمومی حذف شد:** در ابتدا یک fallback به مسیر عمومی قدیمی (`freeconfigs`) نگه داشته شده بود، اما در بازبینی نهایی کاربر خواست این مفهوم کاملاً از بین بره. راه‌حل: هر لینک عمومی قدیمی به‌صورت خودکار به یک **کاربر نامحدود قابل‌حذف** مهاجرت می‌کنه (موقع بالا اومدن اپ) و ردیف قدیمی پاک می‌شه. از این به بعد هر لینک = یک کاربر، بدون استثنا.

## ۳. چیزی که ساخته شد (بک‌اند)

| فایل | تغییر |
|---|---|
| `database.py` | جدول `users` جدید (۱۶ ستون: نام، path، status، duration_days، activated_at، expire_at، paused_at، note، max_devices، last_seen، last_ip، last_user_agent، ...)؛ منطق مهاجرت مسیرهای قدیمی به کاربر |
| `services/user_service.py` (فایل جدید) | کل منطق کاربر: CRUD، اعتبارسنجی path (regex جدا، یکتایی روی دو جدول)، فعال‌سازی اتمیک، Pause/Resume با حفظ زمان، تمدید مدت (با/بدون فعال‌سازی قبلی)، محاسبه‌ی وضعیت مؤثر، تاریخچه‌ی استفاده |
| `services/subscription_service.py` | تابع `generate_dummy_content()` برای کانفیگ ساختگی هنگام انقضا/توقف — با احترام به تنظیم base64/plain موجود |
| `routes/client.py` | روت `/sub/<path>` بازنویسی شد: فقط بر اساس کاربر تصمیم می‌گیره (فعال→سرو واقعی، منقضی/متوقف→dummy، غیرفعال→۴۰۴، ناشناس→۴۰۴) |
| `routes/admin_api.py` | API کامل کاربران: `GET/POST /adminpanel/api/users`، `PUT/DELETE .../<id>`، `/toggle`، `/pause`، `/resume`، `/reset`، `/history` |
| `utils/constants.py` | regex و ثابت‌های جدید برای path و status کاربر |

## ۴. چیزی که ساخته شد (فرانت‌اند)

### بازطراحی کامل ظاهر
- پنل قبلی (تک‌اسکرول شلوغ) به یک ساختار تب‌محور تبدیل شد، و بعد تصمیم گرفتیم کلاً UI رو از نو با ظاهر خوشگل‌تر بسازیم:
- **سایدبار کناری** به‌جای نوار تب بالا
- **تم تیره‌ی مدرن به‌عنوان پیش‌فرض** (با امکان تغییر به روشن، ذخیره در localStorage)
- **فونت Vazirmatn** برای فارسی
- کارت‌ها، جدول‌ها، دکمه‌ها، بج‌ها، فرم‌ها همه با سیستم طراحی جدید (توکن‌های CSS، رنگ‌های semantic) بازسازی شدن

### تب‌های نهایی پنل
۱. **داشبورد** — آمار کلی، نمودارها
۲. **کاربران** — جدول کامل با: نام، لینک (کلیک=کپی)، مدت، انقضا، باقیمانده، وضعیت رنگی، دکمه‌های ویرایش/توقف/ازسرگیری/ریست/فعال‌غیرفعال/حذف/**تاریخچه**
۳. **کانفیگ‌ها** — افزودن و مدیریت کانفیگ‌های استخر سراسری
۴. **اسکن خودکار** — موتور V2RayDAR (وضعیت+تاریخچه و منابع خودکار در یک صفحه یکپارچه، بدون زیرتب داخلی)
۵. **تنظیمات** — فرمت خروجی، ترتیب نمایش، تنظیمات اسکن خودکار (لینک عمومی از اینجا حذف شد)
۶. **لاگ‌ها** — تاریخچه‌ی کامل بازدید لینک‌ها

### قابلیت تاریخچه‌ی کاربر (درخواست بعدی)
برای هر کاربر یک دکمه‌ی 👁️ اضافه شد که مودالی باز می‌کنه با:
- آخرین IP، آخرین User-Agent، آخرین زمان اتصال
- لیست دستگاه‌های متمایز (User-Agentهای مختلف با تعداد دفعات)
- تاریخچه‌ی کامل دسترسی‌ها (زمان، IP، User-Agent، وضعیت)

## ۵. مسیر کار (چون چند بار مسیر عوض شد)

1. **فاز ۰-۳ (بک‌اند):** schema، سرویس کاربر، اتصال روت سابسکریپشن، API — روی یک پوسته‌ی تب‌بالا ساده.
2. **فاز ۴-۵ (فرانت اولیه):** بازچینش تب‌محور پنل موجود، بعد UI واقعی تب کاربران.
3. **بازطراحی کامل:** کاربر خواست ظاهر «خوشگل‌تر» — رفتیم روی یک برنچ جدا (`ui-redesign`)، کل ظاهر (سایدبار+تم تیره+فونت) از نو ساخته شد، بعد merge به `master`.
4. **فاز ۶ (تست):** ۱۲ تست یکپارچه برای مدیریت کاربران اضافه شد؛ مجموع تست‌ها ۵۵ شد.
5. **رفع باگ‌ها:**
   - یک باگ از‌پیش‌موجود (نمودار «آمار بازدید» اصلاً canvas نداشت) پیدا و رفع شد.
   - یک باگ CSS ظریف در تم روشن (متغیرهای legacy در حالت روشن رنگ تیره می‌گرفتن) رفع شد.
6. **اصلاحات نهایی بر اساس بازخورد دقیق‌تر کاربر:**
   - لینک عمومی کاملاً حذف و به کاربر مهاجرت شد (نه فقط مخفی از UI، بلکه واقعاً بازنشسته).
   - مدت و حداکثر دستگاه: `0` = نامحدود.
   - قابلیت تاریخچه/User-Agent هر کاربر اضافه شد.
   - تنظیمات اسکن خودکار به تب تنظیمات منتقل و دو زیرتب داخلی یکی شدن.

## ۶. تست و کیفیت

- **۴۹ تست یکپارچه** (unittest، `test_integration.py`) — شامل auth، CRUD کاربر، یکتایی مسیر روی دو جدول، فعال‌سازی، سرو در همه‌ی حالات (فعال/منقضی/متوقف/غیرفعال)، احترام به فرمت خروجی، مهاجرت، نامحدود، تاریخچه.
- تمام تغییرات فرانت با **تست زنده در مرورگر واقعی** (باز کردن اپ، اجرای عملیات واقعی از UI، خواندن state) تأیید شدن — نه فقط بازرسی کد.
- یک محدودیت محیطی مستند شد: مرورگر تعبیه‌شده‌ی این ابزار نمی‌تونه screenshot بگیره یا مقادیر computed style رو قابل‌اعتماد بخونه؛ برای اثبات ظاهر، یک پیش‌نمایش استاتیک جدا (Artifact) با همون توکن‌های دقیق طراحی ساخته شد.

## ۸. نحوه‌ی آپدیت روی سرور

پروژه یک اسکریپت نصب/آپدیت یک‌خطی (`install.sh`) داره که:
- کد رو از شاخه‌ی `master` گیت‌هاب می‌کشه،
- کل دایرکتوری‌های `services/`, `routes/`, `templates/`, `utils/` رو کپی می‌کنه (پس فایل جدید `user_service.py` هم میاد)،
- سرویس رو stop/start می‌کنه (نه فقط reload)،
- **`database.db` موجود رو دست نمی‌زنه** — جدول `users` خودش موقع بالا اومدن اپ (`init_db()`) ساخته می‌شه، بدون نیاز به migration دستی.

نتیجه: همون دستور نصب/آپدیت یک‌خطی که از قبل استفاده می‌شد، برای اعمال همه‌ی این تغییرات کافیه.

## ۹. نکات باقیمانده / پیگیری بعدی

- `max_devices` فقط ذخیره می‌شه، هنوز enforce نمی‌شه (تصمیم آگاهانه — به دلیل غیرقابل‌اعتماد بودن تشخیص دستگاه در یک aggregator).
- تست تاریخچه‌ی User-Agent در مرورگر واقعی محدودیت داشت (fetch مرورگر نمی‌تونه UA رو جعل کنه)؛ پوشش کامل این رفتار در تست‌های unittest (با کنترل کامل هدرها) انجام شده، نه در تست مرورگری دستی.
