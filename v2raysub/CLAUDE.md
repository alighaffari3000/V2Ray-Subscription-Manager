# دستورالعمل‌های این پروژه برای Claude

## نسخه‌گذاری — الزامی روی هر کامیت رفتاری

این پروژه از Semantic Versioning پیروی می‌کند؛ فرآیند کامل در [VERSIONING.md](VERSIONING.md) است.
فایل [VERSION](VERSION) تنها منبع حقیقت است.

**قانون:** هر کامیتی که رفتار برنامه را تغییر می‌دهد (فیچر، رفع باگ، تغییر رفتار موجود)
باید همراهش نسخه هم bump شود. کامیت‌های صرفاً مستندات/کامنت/refactor بدون تغییر رفتار
نیازی به bump ندارند.

قبل از هر `git commit` روی کد این پروژه:

1. تشخیص بده نوع تغییر چیست (طبق جدول VERSIONING.md):
   - رفع باگ سازگار → PATCH (`1.0.0 → 1.0.1`)
   - قابلیت جدید سازگار → MINOR (`1.0.0 → 1.1.0`)
   - تغییر ناسازگار (breaking) → MAJOR (`1.0.0 → 2.0.0`)
2. مقدار [VERSION](VERSION) را به نسخه‌ی جدید تغییر بده.
3. یک ورودی زیر `## [Unreleased]` در [CHANGELOG.md](CHANGELOG.md) اضافه کن که خلاصه‌ی
   تغییر را توضیح دهد (یا اگر می‌خواهی همان لحظه رسمی منتشر شود، طبق مراحل VERSIONING.md
   سرفصل نسخه‌ی جدید با تاریخ بساز و Unreleased را خالی کن).
4. هر دو فایل (`VERSION` و `CHANGELOG.md`) را همراه با تغییرات کد در همان کامیت stage کن.

اگر مطمئن نیستی نوع تغییر PATCH است یا MINOR (یا اصلاً نیاز به bump دارد یا نه)، از کاربر بپرس —
حدس نزن.

**تگ‌زدن و پوش — الزامی، بدون استثنا:** هر بار که کامیت را به `origin` پوش می‌کنی، باید همان
لحظه نسخه را هم رسماً منتشر کنی و تگش را پوش کنی. **پوش بدون تگ ممنوع است.**

وقتی کاربر می‌گوید «کامیت کن و پوش کن» (یا مشابهش)، این مراحل را کامل انجام بده، نه فقط کامیت/پوش کد:

1. مطمئن شو [VERSION](VERSION) از قبل bump شده (طبق بخش بالا).
2. در [CHANGELOG.md](CHANGELOG.md) ورودی‌های زیر `## [Unreleased]` را به یک سرفصل جدید
   `## [X.Y.Z] - YYYY-MM-DD` (تاریخ امروز) منتقل کن و `## [Unreleased]` را خالی نگه دار.
   لینک‌های compare/release در پایین فایل را هم آپدیت کن.
3. کامیت کن (مثلاً `chore(release): vX.Y.Z` اگر این کامیت فقط همین کار را انجام می‌دهد، یا
   تغییرهای کد + release-prep را در یک کامیت با پیام مناسب).
4. `git push` کد را بزن.
5. تگ بساز و پوش کن:
   ```bash
   git tag -a vX.Y.Z -m "vX.Y.Z"
   git push origin --tags
   ```

این یعنی دیگر نیازی نیست برای «تگ‌زدن» جداگانه تأیید بگیری — تأییدِ «پوش کن» خودش شامل تگ هم می‌شود.
تنها زمانی که باید صبر کنی و بپرسی، وقتی است که نوع نسخه (PATCH/MINOR/MAJOR) نامشخص باشد.
