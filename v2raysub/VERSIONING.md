# فرآیند نسخه‌گذاری (Versioning)

این پروژه از [Semantic Versioning](https://semver.org) با قالب `MAJOR.MINOR.PATCH` استفاده می‌کند.

## تنها منبع حقیقت

نسخه فقط در یک جا نگه‌داری می‌شود:

- فایل [`VERSION`](VERSION) — مثلاً `1.0.0`.

هم برنامه (`config.APP_VERSION` / `config.__version__`) و هم اسکریپت نصب
(`install.sh`) نسخه را از همین فایل می‌خوانند. هیچ‌جای دیگری نباید شماره‌ی
نسخه را دستی بنویسید.

## قانون افزایش نسخه

با توجه به نوع تغییر، یکی از سه بخش را زیاد کنید:

| نوع تغییر | بخشی که زیاد می‌شود | مثال |
|-----------|--------------------|------|
| تغییر ناسازگار (breaking) | MAJOR | `1.4.2 → 2.0.0` |
| قابلیت جدید سازگار | MINOR | `1.4.2 → 1.5.0` |
| رفع باگ سازگار | PATCH | `1.4.2 → 1.4.3` |

هنگام زیاد کردن یک بخش، بخش‌های سمت راست آن صفر می‌شوند.

## مراحل انتشار یک نسخه‌ی جدید

1. مقدار داخل فایل `VERSION` را به نسخه‌ی جدید تغییر دهید.
2. تغییرها را در `CHANGELOG.md` زیر یک سرفصل نسخه‌ی جدید (با تاریخ) منتقل کنید
   و بخش `Unreleased` را خالی بگذارید.
3. کامیت کنید، مثلاً: `chore(release): v1.1.0`.
4. تگ بزنید و منتشر کنید:

   ```bash
   git tag -a v1.1.0 -m "v1.1.0"
   git push origin master --tags
   ```

تگ‌های گیت با پیشوند `v` نوشته می‌شوند (`v1.1.0`)، ولی داخل فایل `VERSION`
پیشوند `v` ندارد (`1.1.0`).
